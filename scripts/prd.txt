# Overview
This project outlines the refactoring of the MagicChecklists WordPress plugin's frontend. The goal is to modernize the development workflow and user interface by adopting Svelte 5 (with Runes), Tailwind CSS 4, and the Flowbite/Flowbite-Svelte component library. The existing PHP backend, including the core WordPress integration and licensing logic, will be preserved. A Vite development server (on port 3005) with Hot Module Replacement (HMR) will be implemented to streamline frontend development, allowing the plugin to load assets directly from Vite during development.

This refactor aims to:
- Improve developer experience with modern frontend tools.
- Enhance UI/UX with a contemporary component library and styling.
- Increase frontend development speed and maintainability.
- Ensure continued compatibility with WordPress and existing PHP backend logic.

# Core Features (of the Refactor Project)
1.  **Svelte 5 Frontend:**
    *   All admin and public-facing UIs (currently PHP/HTML/JS) will be rebuilt using Svelte 5 components.
    *   Leverage Svelte 5's Runes for fine-grained reactivity and improved performance.
2.  **Vite Development Environment:**
    *   Setup Vite as the frontend build tool and dev server.
    *   Configure Vite to run on port 3005 with HMR for Svelte components and Tailwind CSS.
    *   WordPress plugin will conditionally load assets from `http://localhost:3005` during development (e.g., via a `MCL_DEV_MODE` constant).
3.  **Tailwind CSS 4 Integration:**
    *   Utilize Tailwind CSS 4 for utility-first styling across all Svelte components.
4.  **Flowbite & Flowbite-Svelte UI Components:**
    *   Integrate Flowbite (CSS/JS components) and Flowbite-Svelte (native Svelte components) for rapid UI development, ensuring consistency and a modern look.
5.  **Preserved PHP Backend & REST API Interaction:**
    *   The existing PHP backend, including all WordPress hooks, custom post types (`mcl_checklist`), settings (`MCL_Settings`), and REST API controllers (`MCL_RestController`), will remain the primary interface for data and logic.
    *   Svelte components will interact with the backend primarily through the existing REST API endpoints.
6.  **Unaltered Licensing Logic:**
    *   The SureCart licensing PHP logic (`SureCart\Licensing\Client`) will remain untouched.
    *   The UI for license management can be rebuilt with Svelte, but it must interact with the existing PHP licensing mechanisms.
7.  **Production Build Process:**
    *   Vite will be configured to produce optimized, static assets (JS, CSS) for production.
    *   The PHP plugin will load these built assets when not in development mode.

# User Experience (Developer & End-User)
-   **Developer Experience (DX):**
    *   Significant improvement due to Vite's speed, HMR, and the component-based architecture of Svelte.
    *   Simplified styling with Tailwind CSS and pre-built components from Flowbite.
    *   Clear separation of frontend (Svelte) and backend (PHP) concerns.
-   **End-User Experience (UX):**
    *   Potential for a more modern, responsive, and interactive UI for both admin and public-facing checklist interactions.
    *   Faster load times and smoother transitions due to Svelte's performance characteristics.

# Technical Architecture

## 1. Frontend Stack
    -   **Framework:** Svelte 5 (Runes)
    -   **Build Tool/Dev Server:** Vite (configured for port 3005)
    -   **Styling:** Tailwind CSS 4
    -   **UI Components:** Flowbite, Flowbite-Svelte
    -   **Language:** JavaScript/TypeScript (for Svelte components)

## 2. Project Structure (Proposed)
    -   `magicchecklists/` (plugin root)
        -   `package.json`
        -   `vite.config.js`
        -   `tailwind.config.js`
        -   `postcss.config.js`
        -   `src/` (Svelte source files)
            -   `admin/`
                -   `main.js` (entry point for admin Svelte app)
                -   `App.svelte` (root admin Svelte component)
                -   `components/` (shared admin UI components)
                -   `pages/` (Svelte components for different admin screens, e.g., Settings, ChecklistEditor, LicenseManagement)
            -   `public/`
                -   `main.js` (entry point for public Svelte app, e.g., for shortcodes)
                -   `App.svelte` (root public Svelte component)
                -   `components/`
            -   `lib/` (shared utilities, stores, API clients)
        -   `includes/` (existing PHP files - largely unchanged)
        -   `admin/` (existing PHP files for admin enqueueing, page structure - modified to support Svelte apps)
        -   `public/` (existing PHP files for public enqueueing, shortcodes - modified to support Svelte apps)
        -   `dist/` (production build output from Vite, gitignored)
        -   `magicchecklists.php` (modified for dev/prod asset loading)

## 3. PHP Plugin Modifications
    -   **Asset Enqueueing:**
        -   Modify `MCL_Admin` and `MCL_Public` (and potentially `magicchecklists.php`) to conditionally enqueue assets.
        -   If `MCL_DEV_MODE` is true:
            -   Enqueue JS from `http://localhost:3005/src/admin/main.js` (and public equivalent). Vite handles CSS injection.
        -   If `MCL_DEV_MODE` is false:
            -   Enqueue built JS and CSS from the `dist/` directory. Vite will generate a manifest file to get hashed filenames.
    -   **HTML Structure:**
        -   PHP will still render the basic HTML page structure for admin pages and shortcode outputs.
        -   These pages will include a root DOM element (e.g., `<div id="mcl-admin-app"></div>`) where the Svelte app will be mounted.
    -   **Data Localization/Nonces:**
        -   Continue using `wp_localize_script` if necessary to provide initial data (e.g., REST API URL, nonces, current user data) to Svelte apps. Alternatively, fetch this data via an initial REST API call from Svelte.

## 4. Interaction Model
    -   Svelte components will primarily fetch data from and submit data to the existing WordPress REST API endpoints defined in `class-mcl-rest-controller.php`.
    -   AJAX actions in `class-mcl-notification-ajax-manager.php` may need to be refactored to use the REST API if they are driving UI elements that will be converted to Svelte.

## 5. Build Process
    -   **Development:** Run `npm run dev` (or `vite`) to start the Vite dev server.
    -   **Production:** Run `npm run build` (or `vite build`) to generate optimized static assets in the `dist/` folder.

# Development Roadmap

## Phase 1: Core Frontend Setup & Proof of Concept (PoC)
    1.  **Initialize Node.js Project:** `package.json` setup.
    2.  **Install Dependencies:** Vite, Svelte 5, Tailwind CSS 4, Flowbite, Flowbite-Svelte, PostCSS, Autoprefixer, and relevant Vite/Svelte plugins.
    3.  **Vite Configuration:**
        -   Set up `vite.config.js` for Svelte and Tailwind.
        -   Configure dev server for port 3005 and HMR.
        -   Configure build output to `dist/` with manifest generation.
    4.  **Tailwind Configuration:** Set up `tailwind.config.js` and `postcss.config.js`. Integrate Flowbite.
    5.  **PHP Integration for Dev Mode:**
        -   Implement `MCL_DEV_MODE` constant in `magicchecklists.php`.
        -   Modify asset enqueueing logic in PHP to load from Vite dev server when `MCL_DEV_MODE` is true.
    6.  **PoC - Simple Admin Page:**
        -   Choose a simple, non-critical admin UI element or a small settings section.
        -   Create a basic Svelte component for it.
        -   Mount this Svelte component onto a placeholder in the existing PHP-rendered admin page.
        -   Ensure HMR works.
        -   Test basic data fetching from the REST API if applicable.

## Phase 2: Refactor Core Admin Interfaces
    1.  **Checklist CPT Screens:**
        -   Refactor the "All Checklists" list table.
        -   Refactor the checklist editor (metaboxes, task management UI) using Svelte components. This will be the most complex part.
        -   Focus on component reusability for tasks, items, etc.
    2.  **Settings Page (`MCL_Settings`):**
        -   Rebuild the plugin settings page section by section using Svelte and Flowbite components.
    3.  **License Management UI:**
        -   Design and implement a new Svelte-based UI for license management.
        -   This UI must interact with the existing `SureCart\Licensing\Client` PHP methods for activation, deactivation, and status checks, likely via new dedicated REST API endpoints or AJAX actions that wrap the SureCart client calls.
    4.  **Other Admin Pages/UI Elements:** Systematically convert remaining admin UIs.

## Phase 3: Refactor Public-Facing Components
    1.  **Shortcode Output (`MCL_Shortcode`):**
        -   If shortcodes render complex UIs, rebuild these using Svelte components.
        -   The PHP shortcode handler will provide the root HTML element and potentially initial data.
    2.  **Other Public UI Elements:** Address any other frontend elements managed by `MCL_Public`.

## Phase 4: Build Finalization, Testing & Documentation
    1.  **Production Build Testing:** Thoroughly test the production build (`vite build`) and ensure assets are correctly loaded and optimized.
    2.  **Cross-Browser/Device Testing:** Verify UI consistency and functionality.
    3.  **WordPress Compatibility Testing:** Test with different WordPress versions and common plugin configurations.
    4.  **Performance Optimization:** Analyze frontend performance and optimize Svelte components and asset loading.
    5.  **Developer Documentation:** Document the new frontend architecture, build process, and how to create/modify Svelte components.

# Logical Dependency Chain
1.  **Foundation:** Phase 1 (Core Frontend Setup & PoC) is foundational for all subsequent work.
2.  **Backend Stability:** The existing PHP REST API must be stable and well-understood before extensive frontend work.
3.  **Admin Core First:** Refactoring admin interfaces, particularly the checklist editor, is a high priority as it's core to the plugin's functionality.
4.  **Component Reusability:** Design Svelte components with reusability in mind (e.g., task item, input fields) to accelerate development.
5.  **Iterative Refactoring:** Approach each admin page or feature as an incremental step.

# Risks and Mitigations

1.  **Complexity of Integrating Vite with WordPress:**
    *   **Risk:** Ensuring seamless asset loading in both dev (Vite server) and production (static build) modes within the WordPress enqueueing system can be tricky.
    *   **Mitigation:** Clear PHP logic based on `MCL_DEV_MODE`. Thoroughly test Vite's manifest generation and PHP's ability to parse it for production assets. Start with a minimal PoC for asset loading.
2.  **Learning Curve (Svelte 5, Runes, Tailwind 4):**
    *   **Risk:** If the team is new to these technologies, development might be slower initially.
    *   **Mitigation:** Allocate time for learning. Start with simpler components. Utilize official documentation and community resources. Pair programming can help disseminate knowledge.
3.  **Refactoring Complex UI (e.g., Checklist Editor):**
    *   **Risk:** The checklist editor likely has complex state management and interactivity. Replicating this in Svelte could be challenging.
    *   **Mitigation:** Break down the editor into smaller, manageable Svelte components. Plan state management carefully (Svelte stores). Refer extensively to the existing JS logic.
4.  **SureCart Licensing UI:**
    *   **Risk:** The existing licensing page is added by the SureCart client. Directly modifying its output might be difficult or unsupported.
    *   **Mitigation:**
        *   Option A (Preferred): Create a *new* Svelte-powered admin page for licensing. This page will make REST API calls to new PHP endpoints that, in turn, call the necessary `SureCart\Licensing\Client` methods (e.g., `activate_license()`, `deactivate_license()`, `get_license()`).
        *   Option B: Investigate if the SureCart client offers hooks or filters to customize the settings page output. This is less likely to offer full control for a Svelte app.
        *   Option C (Fallback): Leave the existing SureCart-generated licensing page as-is if refactoring its UI is too complex or risky, focusing on other areas first.
5.  **Scope Creep:**
    *   **Risk:** The desire to improve UI/UX significantly during the refactor might lead to scope creep beyond a 1:1 feature replacement.
    *   **Mitigation:** Prioritize functional parity first. Log desired enhancements as separate tasks/features to be addressed after the core refactor is complete.
6.  **Maintaining PHP Backend Interaction:**
    *   **Risk:** Ensuring all necessary data is available via the REST API and that Svelte correctly interacts with it.
    *   **Mitigation:** Review existing REST endpoints. Add new ones or modify existing ones if Svelte components require different data structures or actions. Implement robust error handling in Svelte API clients.
